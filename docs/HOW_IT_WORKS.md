## What This Document Covers

A public-facing description of the game's technical systems as they exist in the current prototype. This covers how the game runs, how the input works, what the player sees, how the software captures breathing data, and why the systems are designed the way they are. Script names are included throughout so this document can serve as a reference while browsing the codebase.

All scripts live under `BREATHE_Game/Assets/Scripts/`, organized into five assemblies:

- **Breathe.Utility** (`Scripts/Utility/`) — Math helpers, converters, animation utilities
- **Breathe.Data** (`Scripts/Data/`) — ScriptableObject configs and data definitions
- **Breathe.Input** (`Scripts/Input/`) — Breath input interface and implementations
- **Breathe.Gameplay** (`Scripts/Gameplay/`) — All game logic, AI, course, environment, overlays
- **Breathe.UI** (`Scripts/UI/`) — Menu screens, HUD, celebration display

Dependencies flow strictly downward — UI depends on Gameplay, Gameplay depends on Input, Input depends on Data and Utility, but never the reverse. This is enforced by Unity's assembly definition system. The reason for this strict layering is compile isolation: a change to a UI script cannot break the input layer, and a change to a data config cannot break utility functions. It also prevents accidental coupling between systems that shouldn't know about each other, and speeds up incremental compilation since only affected assemblies recompile.

## The Big Picture

The player blows toward a custom-built fan device. The fan's rotation is measured by a microcontroller and transmitted to the game over USB. Inside the game, that signal becomes wind that fills a sailboat's sail and pushes it through a race course alongside AI opponents. The player always finishes. Every session ends with a celebration of personal effort and a display of breathing data.

**breath → fan spin → wind power (game mechanic power variable) → sail speed.**

The core design philosophy is **one input, zero failure.** Breath is the only control the player has — no buttons, no steering, no secondary inputs. The boat navigates itself; the player just controls how fast it goes. This keeps the cognitive load minimal so the player can focus entirely on the act of breathing. And because the game guarantees a finish regardless of effort level, there's never a reason to stop playing — which is critical for both engagement and data collection.

## Procedural Runtime Generation

The Unity scene is intentionally minimal. The editor contains only three visible objects — the player's boat and two AI boats. Everything else the player sees is generated by the scripts at runtime:

- **Ocean surface** — Three animated sine-wave layers (surface, depth, foam) with parallax scrolling, attempting to create a stylized Wind Waker-inspired ocean. Built by `ScrollingOceanBackground.cs`, producing ~106 LineRenderers with multi-harmonic wave composition.
- **Race course** — Buoy lanes (red/green pairs), center dashes, distance rings, and a checkered finish line. Course shape is randomly selected each run from a pool of named layouts defined in `CourseLayout.cs` assets. Built by `CourseMarkers.cs`. Buoys bob on the water with sinusoidal animation and Perlin noise jitter.
- **Waypoints** — The invisible path boats follow, generated along the same mathematical curve as the buoys by `CourseManager.cs` calling into `CourseMarkers.CurveX()` so everything aligns.
- **Obstacles** — Rocks that spawn in AI lanes as the race progresses. The player's path stays clear. Built and object-pooled by `CourseObstacleSpawner.cs`. AI boats get stunned on collision via `ObstacleCollision.cs`.
- **Environmental zones** — Wind hazards (headwinds, tailwinds, crosswinds, cyclones, doldrums, choppy water) placed along the course. Each type has distinct visual particles and physics effects. Spawned and object-pooled by `EnvironmentalZoneSpawner.cs`, visuals handled by `ZoneVisualController.cs`, physics by `EnvironmentalZoneEffect.cs`.
- **Background scenery** — Decorative islands outside the course boundaries, spawned and recycled by `ScenerySpawner.cs` as the camera moves.
- **Boat visual effects** — Wind streaks (`BoatWindEffect.cs`), bow splashes (`BoatSplashEffect.cs`), and wake trails (`BoatWakeTrailEffect.cs`) generated per boat and scaled by speed.
- **UI overlays** — Countdown, tutorial, zone notifications, debug telemetry, and result screens — all created procedurally by `CourseManager.EnsureOverlays()` if they don't already exist in the scene.

**Why build everything procedurally?** Three reasons drove this decision:

1. **Scene maintainability.** With runtime generation, the Unity scene stays minimal — just three boats and a handful of manager objects. There are no sprawling prefab hierarchies, no manually positioned markers, no fragile reference chains to break when something changes. The managers produce what they need at runtime, which means the scene is easy to understand, easy to modify, and unlikely to end up in a broken state.

2. **Replayability.** Because the course layout, obstacle positions, and zone placements are generated fresh each run, no two races are identical. This matters for the therapeutic use case: if the player is doing breathing exercises through gameplay, variety across sessions prevents habituation and keeps engagement high.

3. **Prototype speed.** Generating visuals from code eliminates the art pipeline as a bottleneck. The game can be developed, tested, and demonstrated without waiting for art assets. When a production art pass happens later, the code infrastructure stays identical — sprites and particle systems can accept imported assets without script changes.

Obstacles, zones, and scenery use **object pooling** (pre-allocating a fixed number and recycling them as the camera scrolls) rather than creating and destroying objects during gameplay. This avoids garbage collection spikes that would cause frame rate hitches — important for a game targeting accessibility, where visual stuttering can be disorienting.

## Breath Input Pipeline

The game reads breath intensity through a common interface (`IBreathInput.cs`). Three input sources are supported, all producing the same normalized 0-to-1 signal. The active source is managed by `BreathInputManager.cs`, a singleton that exposes whichever input mode is selected. Gameplay code reads from this manager and never knows which source is active.

**Why a common interface?** The game must work in three very different contexts: with the custom fan hardware (the target experience), with a laptop microphone (for testing or demos without hardware), and with keyboard/gamepad input (for development). Rather than scattering conditional logic throughout the codebase (`if hardware then X, else if mic then Y`), a single interface means every gameplay system — wind, scoring, analytics, AI comparison — reads from one place. The input source can be swapped at runtime from the main menu without restarting the game, and adding a new input device in the future means writing one new class that implements the interface.

**Custom hardware** (`FanBreathInput.cs`) — A DC hobby motor with a propeller acts as a generator. When the player blows, the propeller spins and produces voltage proportional to effort. An Arduino microcontroller reads this voltage power level change and transmits it to the game over USB serial. A background thread in the game reads serial data continuously so the main game loop is never blocked. The background thread exists because serial reads are blocking I/O — without it, every serial read would freeze the game for milliseconds, causing visible stuttering.

**Microphone fallback** (`MicBreathInput.cs`) — Any laptop or desktop microphone captures breath sound. The software isolates low-frequency energy (breath is low-frequency, voice is higher) and maps the amplitude to intensity. The low-frequency bias is intentional: it distinguishes actual breath from talking, ambient noise, or other non-breath sounds. A calibration step captures the player's baseline noise level and maximum effort to personalize the mapping, because microphone sensitivity and ambient noise vary enormously between environments.

**Simulated input** (`SimulatedBreathInput.cs`) — For development and demos without hardware. Keyboard or gamepad input ramps intensity up and down with configurable speed, mimicking the feel of breathing. The ramp-up and decay are tuned to feel like actual breath effort rather than an instant on/off switch, so developers can test how the game feels without needing the physical device.

All three sources pass through the same processing pipeline using shared functions from `SignalProcessing.cs`:

- **Range normalization** — Maps the raw signal (RPM, RMS amplitude, or keyboard ramp value) into a 0-to-1 range so downstream systems never need to know the signal's native units
- **EMA smoothing** — Exponential moving average removes frame-to-frame jitter while staying responsive to real changes. Chosen over simple averaging because EMA is frame-rate independent (the math uses `Pow` with delta time) and naturally weights recent samples more heavily
- **Dead zone** — Zeros out values below a configurable threshold. Without this, electrical noise from the sensor or ambient room sound from the microphone would register as faint breath, causing the boat to creep forward when the player is at rest
- **Clamping** — Final output is clamped to 0–1 so every consumer can assume a bounded range

Tuning parameters (smoothing factor, dead zone threshold, power level thresholds, ramp/decay speeds) are stored in a `BreathConfig.cs` ScriptableObject asset — editable in the Unity Inspector without code changes.

## Wind and Propulsion

Breath intensity feeds into `WindSystem.cs`, which applies additional EMA smoothing and a response curve exponent. The response curve uses an exponent less than 1.0, which compresses the lower end of the input range — meaning light breath produces a proportionally larger effect than the raw value would suggest. This is a deliberate accessibility decision: the target audience includes players with limited lung capacity, and making small effort feel meaningful keeps the experience rewarding even when the player can't blow hard.

Wind power drives the player's boat via `SailboatController.cs`:

**Boat speed = base speed + (wind power × breath bonus multiplier) × environmental modifier**

The boat always moves at base speed even with no breath input — the player is never stuck. This is a core design rule. "Dead time" where nothing happens on screen is the fastest way to lose a player's attention, especially a young or easily frustrated one. Breath adds speed on top, so more effort always means more reward, but zero effort still means forward progress.

Environmental zones modify the multiplier (headwinds reduce it, tailwinds increase it), creating natural variation in how much effort is needed at different points in the race.

The sail visually inflates and leans in proportion to wind power. At rest, the sail is thin and upright. At full power, it billows out and tilts. This visual feedback is critical because with only one input and no steering, the player needs to see that their breath is doing something. The sail is the primary feedback channel — it makes the connection between physical blowing and in-game effect viscerally obvious.

The boat auto-navigates the course via waypoint following — breath is the only player control, acting purely as the accelerator. There is no manual steering. This decision was made for three reasons: it keeps the input model to a single dimension (effort intensity), it reduces cognitive load so the player focuses on breathing rather than navigation, and it ensures the game is accessible to players who may have limited motor control beyond the ability to blow.

Speed and multiplier tuning live in `CourseConfig.cs` (base speed, breath bonus multiplier, soft time cap) — editable in the Unity Inspector without code changes.

## Race Structure

A race follows a fixed sequence, orchestrated by `GameStateManager.cs` (the top-level state machine) and `CourseManager.cs` (race lifecycle):

1. **Menu** (`MainMenuUI.cs`) — Player selects input mode and starts a race. Quick Play option skips level select.
2. **Level Select** (`LevelSelectUI.cs`) — 2×3 grid of minigame cards read from `MinigameManager.cs`. Currently shows only SAILBOAT. Tapping a card selects the minigame.
3. **Calibration** (`CalibrationUI.cs`) — Microphone input runs a two-phase calibration (baseline + max effort). Hardware and simulated input skip this step automatically. Calibration exists because microphone sensitivity varies wildly across devices and environments — the same breath that reads as 80% on one laptop might read as 20% on another. The two-phase approach (quiet room baseline, then blow as hard as you can) maps the player's personal range onto the 0–1 scale.
4. **Tutorial** (`TutorialPopup.cs`) — Brief instructional overlay explaining the controls. Game is paused until dismissed.
5. **Countdown** (`CountdownOverlay.cs`) — 3-2-1-Go with scale animation. Boats are visible but frozen during the countdown. Showing the boats during the countdown builds anticipation and gives the player a moment to orient themselves before the race begins.
6. **Race** — Boats move, timer runs, environmental zones and obstacles appear. The player blows to gain speed. HUD elements (`HUDController.cs`) show a wind meter, race progress bar, and zone notifications. The debug overlay (`DebugOverlay.cs`) provides live telemetry during development.
7. **Soft time cap** — If the race exceeds a configurable duration (~60 seconds, set in `CourseConfig`) and the player hasn't finished, `CourseManager` moves the finish line directly ahead of the player. The player always crosses the finish line. This is the ultimate no-fail safety net: no matter what happens — hardware failure, player distraction, extremely low effort — the game ends in a finish. This guarantees that every session produces a complete data record and that the player never experiences a "you lost" screen.
8. **Celebration** — Race stops. Post-race sail-away animation plays (`PostRaceSailAway.cs`), then results appear (`RaceResultOverlay.cs`, `CelebrationScreenUI.cs`) with personal stats and comparisons. The sail-away animation — where AI boats catch up and cruise alongside the player before peeling off — creates an emotional beat between competition and celebration, giving the results overlay time to appear naturally rather than abruptly.

The state machine design (explicit named states with guarded transitions) was chosen over simpler approaches like coroutine chains because it makes the game flow predictable and debuggable. Every system can check `GameStateManager.CurrentState` to know exactly where in the flow the game is, and invalid transitions are caught immediately.

The minigame abstraction is handled through `IMinigame.cs` — a contract that any game mode implements. The sailboat race registers itself as the active minigame. Future minigames plug into the same slot without touching existing code. This interface-based architecture means the celebration screen, scoring, data persistence, and navigation all work generically for any game that implements the contract. Minigame metadata (name, thumbnail, breath pattern) is defined in `MinigameDefinition.cs` ScriptableObject assets.

## AI Companions

Two computer-controlled boats race alongside the player, managed by `AICompanionController.cs`. Each AI boat has a randomized personality — slightly different speed, turning rate, reaction time, and breathing rhythm — so no two races feel identical.

The AI exists to create the feeling of a shared experience rather than a solo exercise. Racing alone against a timer feels clinical. Racing alongside opponents who react, struggle, and occasionally stumble feels like a game. The AI personalities are randomized per race so the player can't predict or memorize their behavior.

AI speed is governed by several factors:
- **Base movement** with per-boat variation from Perlin noise functions — Perlin noise was chosen over simple random values because it produces smooth, natural-looking speed fluctuations rather than jerky random changes
- **Simulated breathing** — AI sails inflate and deflate on a cycle, mimicking a real player's effort pattern. This sells the illusion that the AI boats are also breath-powered, making the race feel consistent rather than obviously rigged.
- **Rubber-banding** — AI boats adjust pace relative to the player. If the player is ahead, they speed up slightly. If behind, they slow down. This keeps the race visually close and exciting regardless of the player's actual effort level. Without rubber-banding, a strong player would leave the AI behind immediately (boring), and a weak player would never see the AI (discouraging).
- **Finish-line slowdown** — As the race nears its end, AI boats progressively decelerate to ensure the player finishes first. This is the "player always wins" guarantee implemented in the AI layer.
- **Periodic hesitation** — Occasional brief slowdowns add natural imperfection. Perfect AI feels robotic; occasional stumbles feel alive.

AI boats can hit obstacles and get stunned — `AIStunEffect.cs` zeros their speed and plays a cartoon wobble/spin animation with orbiting star sprites for 1.5–3 seconds before recovery. Collision detection is handled by `ObstacleCollision.cs` on each rock obstacle. Stun events give the player satisfying moments of overtaking without requiring any skill — just being in the right place while the AI hits a rock.

In rare cases where the player barely participates (very low breathing activity), one AI boat may be allowed to win. This is a deliberate edge case: if the AI always lets the player win even when they clearly aren't trying, the game feels patronizing and fake. Allowing a rare loss preserves the credibility of the competition. The player finishes at worst in second place. All AI tuning parameters (speed variation, rubber-banding strength, stun duration, competitive win thresholds) live in `AIConfig.cs`.

## Environmental Zones

Six types of wind conditions spawn along the course, each creating a different challenge. Zone physics are applied by `EnvironmentalZoneEffect.cs`, which uses trigger colliders to detect boats entering and leaving each zone:

- **Headwind** — Significantly reduces speed and pushes the boat backward. Blow harder to push through.
- **Tailwind** — Free speed boost with no resistance. No escape needed — enjoy the ride.
- **Crosswind** — Normal speed but a strong lateral push to one side. Blow to counter the drift.
- **Cyclone** — Reduced speed with a pull toward the center. If the boat reaches the middle, it spins. Sustained blowing breaks free.
- **Doldrums** — Speed drops to near zero with a gentle sideways drift. Any sustained breathing restarts movement.
- **Choppy Water** — Moderately reduced speed with horizontal wobble. Steady rhythmic breathing stabilizes the boat.

**Why six different zone types?** Each zone demands a different breathing response — hard sustained blowing (headwind), relaxation (tailwind), quick reactive bursts (crosswind), sustained effort (cyclone), gentle sustained breath (doldrums), and steady rhythmic breathing (choppy water). This maps directly to the therapeutic goal: a single race naturally exercises multiple breathing patterns without the player consciously thinking about it. The variety also prevents habituation — the player can't settle into one rhythm for the entire race.

Every zone is escapable within a fixed time limit, even without breathing. Breath effort accelerates the escape. This is a critical design constraint: no zone can permanently trap or punish a player. The game guarantees forward progress at all times. This means zones are challenges that reward effort, not obstacles that gate progress.

Zones are placed by `EnvironmentalZoneSpawner.cs` using weighted random selection (recently-spawned types get lower weight) with enforced minimum spacing and a grace period at the start. The weighted selection prevents the same zone type from appearing repeatedly — if the player just fought through a headwind, the next zone is unlikely to be another headwind. The grace period (no zones near the starting line) lets the player get oriented before encountering hazards. Visual effects (ring particles, color coding) are procedurally created by `ZoneVisualController.cs`. Per-zone-type parameters (speed multiplier, lateral force, escape time, visual color, popup text) are configured in `EnvironmentalZoneConfig.cs` ScriptableObject assets with hardcoded fallback defaults so the game works even if the config asset is missing.

Both the player and AI boats are affected by zones equally — both implement the `IBoatEnvironmentalTarget.cs` interface. This shared interface was chosen so zone code doesn't need separate logic for "is this a player or AI" — it applies the same physics to anything that enters the trigger. The AI then responds to the zone's speed reduction through its own speed calculation, just as the player responds by blowing harder.

## Scoring and Breath Analytics

The game captures two layers of data during every race. The key design decision here is that **measurement is invisible** — the player never enters a "measurement mode" or does anything special. Every metric is computed in real time while they play. Playing the game is the measurement.

### Race Performance (`ScoreManager.cs`)
- Total time the player spent actively breathing
- Peak breath intensity reached during the race
- Longest single sustained blow
- Race completion time
- Number of environmental zones conquered while blowing

Personal bests are stored locally via `PlayerPrefs` and compared on the celebration screen. New records are highlighted. Personal bests serve as intrinsic motivation — the player competes against their own history rather than an external standard, which aligns with therapeutic framing ("you're improving") rather than competitive framing ("you're winning or losing").

### Breath Pattern Analysis (`BreathAnalytics.cs`)
Running alongside the race, a separate analytics system classifies the player's breathing behavior:

- **Sustained segments** — Count and average duration of continuous blows lasting more than 1.5 seconds
- **Bursts** — Count of short, sharp blows under 0.8 seconds
- **Average intensity** — Mean effort across the entire session
- **Activity ratio** — Fraction of time actively breathing vs total session time
- **Breath pattern classification** — Categorized as Sustained, Burst-Dominant, Rhythmic, or Mixed based on segment data
- **Zone response rate** — How often the player adjusted effort appropriately when entering a new environmental zone

These metrics were chosen because they map to recognized categories in respiratory therapy. Sustained breathing capacity, burst control, rhythmic consistency, and adaptive response to changing conditions are all clinically meaningful dimensions of respiratory function. By capturing them passively during gameplay, the system produces data that could inform clinical assessment without requiring a clinical protocol.

### Data Persistence (`SessionLogger.cs`)
Each completed round is saved as a structured NDJSON record in a daily log file (`Application.persistentDataPath/sessions/YYYY-MM-DD.jsonl`). Records include all race stats plus the full breath analytics snapshot. The NDJSON format (one JSON object per line) was chosen because it is append-only and crash-safe — if the game crashes mid-write, only the incomplete final line is lost; all previous records remain intact. It also requires no database, works entirely offline, and can be parsed by any tool that reads JSON. This enables session-over-session tracking without requiring the player to do anything beyond playing the game.

### Safety Monitoring (`BreathSafetyMonitor.cs`)
A safety system monitors prolonged high-effort breathing and suggests rest breaks if the player has been blowing hard for an extended period. This is based on clinical literature (Joo et al. 2015) documenting that breath-based therapy games can cause dizziness without adequate rest periods. The system tracks both continuous high-effort duration and cumulative high-effort time, firing gentle break suggestions through UI events. It exists because a game designed to encourage breathing exercises has an ethical obligation to prevent the player from overexerting — especially given the target audience may include children or patients with respiratory conditions.

## Hardware Signal Path

```
Player breath
  → Fan propeller spins (physical rotation from airflow)
  → DC motor generates voltage proportional to rotation speed
  → Arduino microcontroller reads voltage on analog input (Arduino/BreathFanRPM/BreathFanRPM.ino)
  → Arduino transmits "RPM:####" over USB serial at 9600 baud
  → FanBreathInput.cs reads serial data on a background thread
  → Signal is normalized, smoothed, and mapped to 0–1 intensity (SignalProcessing.cs)
  → BreathInputManager.cs exposes intensity to all game systems
```

The microcontroller sketch and the game's serial reader use a simple text protocol (`RPM:####` lines at ~20 Hz). A simple text protocol was chosen over binary because it is human-readable for debugging (open a serial monitor and see the values), trivially parseable, and robust to partial reads. The 20 Hz update rate provides smooth input at well above the game's frame-rate needs.

The game also supports microphone and simulated input through the same `IBreathInput` interface, so gameplay logic is completely independent of the input source. This independence is what allows the game to function as a complete prototype even before hardware is connected — every system, every calculation, every data point works identically regardless of whether the signal comes from a physical fan, a microphone, or a keyboard.

## Debug and Development Tools

`DebugOverlay.cs` provides a built-in telemetry HUD during gameplay, showing:
- Course layout name and length
- Game state, race timer, progress percentage, placement
- Active input source, breath intensity, power level, wind power
- Player boat speed (in game units, knots via `WindSpeedConverter.cs`, and mph), position, finish status
- Per-AI boat progress and stun state
- Current environmental zone type
- Frame rate and system timing

The overlay uses Unity's `OnGUI` immediate-mode rendering rather than a Canvas-based UI. This was chosen because `OnGUI` requires no scene setup — it renders without any GameObjects, canvases, or prefabs existing in the scene. This makes it resilient: even if the game UI is broken or missing, the debug overlay still works. It's a development tool that can never be broken by changes to the game itself.

The overlay is toggled with a hotkey and cycles between compact, expanded, and minimal display modes. It renders in both the editor and standalone builds.

## Configuration Architecture

All gameplay tuning lives in ScriptableObject assets — Unity's data container pattern for storing configuration separately from the scripts that use it. Every ScriptableObject is editable in the Unity Inspector without code changes and without recompiling.

This separation of data from logic is a deliberate architectural choice: a designer or researcher can adjust breath sensitivity, AI difficulty, zone intensity, race length, or safety thresholds by dragging sliders in the editor. No code knowledge needed, no risk of breaking game logic, and changes take effect immediately in the next play session. Multiple config profiles can exist side-by-side (e.g., one tuned for young children, another for adults) and be swapped by reassigning a single reference.

Config assets in the prototype: `BreathConfig.cs`, `CourseConfig.cs`, `AIConfig.cs`, `EnvironmentalZoneConfig.cs`, `MinigameDefinition.cs`, and `CourseLayout.cs`.

## Design Principles in the Prototype

These principles are enforced by the software, not just by design intent:

- **The player always finishes.** A soft time cap in `CourseManager` moves the finish line to the player if the race runs long. This exists because a game that can be "lost" doesn't work for therapeutic use — the player needs to feel safe and successful every time.
- **Every zone is escapable.** Time alone is sufficient; breath helps escape faster. Guaranteed by `EnvironmentalZoneEffect`'s max escape timer. This ensures challenges create breath variation without creating frustration.
- **AI never permanently outpaces the player.** Rubber-banding and finish slowdown in `AICompanionController` ensure a positive outcome. The rare competitive win exception preserves credibility.
- **The game works without custom hardware.** `MicBreathInput` and `SimulatedBreathInput` are always available as fallbacks. This ensures the game can be demonstrated, tested, and used in any environment.
- **Every stat is captured during gameplay.** No separate measurement mode — `ScoreManager` and `BreathAnalytics` run while the player plays. Playing the game is the measurement. This eliminates compliance barriers in clinical settings.
- **The experience is different every run.** Course layout (`CourseLayout` pool), obstacle positions (`CourseObstacleSpawner`), and zone placements (`EnvironmentalZoneSpawner`) are randomized. Variety prevents habituation and keeps the breathing exercise engaging across sessions.
- **The scene works without prefabs.** Everything is procedurally generated from code. The scene stays minimal and maintainable — managers create what they need at runtime, so there are no fragile reference chains to break.
